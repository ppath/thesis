# SLURM Configuration Fix Plan #2 — `enif` (plast-hpc)

## Context

Round 1 (`0001. fix plan.md`) cleaned up the obvious bugs (password placeholder, missing priority/multifactor, shard `Type=`, archive settings, etc.) — those are now applied. This round is the second pass on the same files plus `setup.sh` and `README.md`. It catches issues that survived the first review and a few omissions that show up only when you cross-reference `slurm.conf` against `gres.conf` and `setup.sh` against `slurmdbd.conf`.

The intended outcome: a config that boots cleanly on first run, doesn't silently degrade on re-run, and doesn't hide NUMA-locality bugs that only surface for socket-1 GPU jobs.

---

## Findings

### CRITICAL

**1. `Epilog=/etc/slurm/epilog.sh` references a file that does not exist** (`slurm.conf:150`)
- The script is not in the repo, `setup.sh` does not create or copy it, and the README does not mention it. Every time a job ends, slurmd will log `error: Epilog failed` (and depending on version may DRAIN the node).
- **Fix**: Pick one:
  - (a) Ship a real `etc/slurm/epilog.sh` (e.g. `nvidia-smi --query-compute-apps=pid --format=csv,noheader | xargs -r kill -9` + a final `nvidia-smi -q -d MEMORY` log line for the allocated GPUs), make it executable, and add a copy step to `setup.sh` that installs it `0755 root:root`.
  - (b) Remove the `Epilog=` line and `EpilogMsgTime=2000` until you actually have a script.

**2. `setup.sh` is not idempotent — re-running destroys the cluster** (`setup.sh:33,41-45,61-63`)
- `create-munge-key -f` overwrites `/etc/munge/munge.key` on the second run, breaking auth between any nodes already configured (and invalidating queued jobs on this one).
- `CREATE DATABASE slurm_acct_db;` and `CREATE USER 'slurm'@'localhost' …` both error on second run; `set -e` (once added — see #11) would abort the script there.
- `sacctmgr -i add cluster plast-hpc` errors if the cluster was auto-registered by slurmctld at first start.
- **Fix**:
  - Guard munge key: `[ -s /etc/munge/munge.key ] || sudo /usr/sbin/create-munge-key -f`.
  - SQL: `CREATE DATABASE IF NOT EXISTS slurm_acct_db; CREATE USER IF NOT EXISTS 'slurm'@'localhost' IDENTIFIED BY '${SLURM_DB_PASS}'; ALTER USER 'slurm'@'localhost' IDENTIFIED BY '${SLURM_DB_PASS}'; GRANT …`.
  - `sacctmgr` calls: append `|| true` (or check existence first via `sacctmgr -nP show cluster | grep -q ^plast-hpc`).

**3. `slurmdbd.conf` declares `ArchiveDir=/var/spool/slurm/archive` but the directory is never created** (`slurmdbd.conf:53`, `setup.sh:7`)
- The `slurmdbd.conf` comment tells the operator to create it; `setup.sh` step 2 does not. First purge run after deployment will log archive errors.
- **Fix**: In `setup.sh` step 2, add `sudo mkdir -p /var/spool/slurm/archive && sudo chown slurm:slurm /var/spool/slurm/archive && sudo chmod 750 /var/spool/slurm/archive`.

### HIGH

**4. `CoreSpecCount=2` reserves cores 30–31 — both on socket 1 — undermining `gres.conf` NUMA locality for GPU2/GPU3** (`slurm.conf:189`, `gres.conf:37-38`)
- SLURM picks spec cores from the highest-numbered abstract cores. With `Sockets=2 CoresPerSocket=16`, cores 30–31 are socket 1. `gres.conf` says GPU2/GPU3 prefer `Cores=16-31`, but only 14 of those are actually schedulable. A job asking `--gres=gpu:h100:1 -c 16` that lands on GPU2/GPU3 either pends or spills cross-socket — silently defeating the locality config.
- **Fix**: Drop `CoreSpecCount=2` and rely on `MemSpecLimit=12288` for OS headroom (32 cores is plenty; the OS does not need 2 dedicated). If you want core reservation, do it symmetrically by reducing `Cores=` in `gres.conf` to `0-14` and `16-30` and not setting `CoreSpecCount` at all — but the simpler fix is to drop it.

**5. `libpmix-dev` is a build-time package; PMIx MPI launches may fail at runtime** (`README.md:11`, `setup.sh:4`, `slurm.conf:115`)
- `MpiDefault=pmix` requires the SLURM PMIx plugin, which on Ubuntu is in `slurm-wlm-basic-plugins` (and the runtime lib is `libpmix2`, pulled in transitively by `-dev`). If the plugin is not installed, `srun --mpi=pmix` fails with `MPI plugin not found`.
- **Fix**: Either
  - (a) Add `slurm-wlm-basic-plugins` to the `apt install` line and verify with `srun --mpi=list` post-install, or
  - (b) If MPI is not actually needed for thesis workloads, set `MpiDefault=none` and drop the `libpmix-dev` package.

**6. `sed -i "s/__SLURM_DB_PASS__/${SLURM_DB_PASS}/g"` is unsafe and leaks the password** (`setup.sh:28`)
- Any password containing `/`, `&`, or `\` corrupts the substitution. The full password also appears in the running `sed` command line, visible to other users via `ps` or `/proc/<pid>/cmdline` for the duration of the call.
- **Fix**: Use a delimiter unlikely to appear in passwords and pass via stdin / heredoc so the password never sits in argv:
  ```bash
  sudo install -m 0600 -o slurm -g slurm /dev/stdin /etc/slurm/slurmdbd.conf < <(
    SLURM_DB_PASS="$SLURM_DB_PASS" envsubst '$SLURM_DB_PASS' < ./etc/slurm/slurmdbd.conf
  )
  ```
  and change the placeholder to `${SLURM_DB_PASS}` (envsubst form). Alternatively use `python3 -c 'import os,sys; sys.stdout.write(sys.stdin.read().replace("__SLURM_DB_PASS__", os.environ["SLURM_DB_PASS"]))'`.

**7. `sleep 2` between `slurmdbd` and `slurmctld` is a race** (`setup.sh:55-57`)
- On a slow boot or busy MariaDB, slurmdbd takes >2 s to be reachable. slurmctld then starts, fails to register, and exits.
- **Fix**: Replace with a real readiness check:
  ```bash
  for _ in $(seq 1 30); do
    ss -ltn 'sport = :6819' | grep -q LISTEN && break
    sleep 1
  done
  ```

### MEDIUM

**8. `InactiveLimit=300` kills interactive sessions after 5 min of idle I/O** (`slurm.conf:147`)
- `srun --pty bash` sessions that sit at a prompt while the user reads a paper get killed. Painful on a single-user research box.
- **Fix**: Set `InactiveLimit=0` (disable) or `1800` (30 min). Round 1 raised this from 0 to 300 for runaway-allocation protection; for thesis interactive use 1800 is a better tradeoff.

**9. `ReturnToService=2` auto-resumes after any DOWN, masking real problems** (`slurm.conf:70`)
- Includes `RealMemory` mismatches and gres failures. A flapping node will keep coming back without anyone noticing the underlying cause.
- **Fix**: `ReturnToService=1` — auto-resume only if not drained-by-sysadmin and slurmd reports healthy.

**10. `AllowedSwapSpace=0` + `ConstrainSwapSpace=yes` OOM-kills jobs that briefly touch swap** (`cgroup.conf:22-23`)
- PyTorch's caching allocator and large NumPy operations occasionally produce transient swap pressure even when peak RSS stays under `--mem`. With swap forbidden, these jobs die abruptly.
- **Fix**: `AllowedSwapSpace=10` absorbs short overshoots without giving real escape (a job genuinely needing more RAM still dies, just less spuriously).

**11. `setup.sh` missing shebang and strict-mode flags** (`setup.sh:1`)
- A failing `sudo cp` mid-script currently keeps going; partial deployments are hard to debug.
- **Fix**: First line `#!/usr/bin/env bash` followed by `set -euo pipefail`.

**12. No `MaxMemPerCPU` on the `gpu` partition** (`slurm.conf:206-213`)
- A user passing `-c 32 --mem=400G` can claim almost all RAM in one job. Single-user today, but worth setting now since it's free.
- **Fix**: Add `MaxMemPerCPU=16384` (16 GB/CPU cap → matches ~512 GB / 32 physical cores).

**13. `ufw allow 6818,6819/tcp` opens ports that should be localhost-only** (`setup.sh:52`)
- 6818 (slurmd) and 6819 (slurmdbd) only need cross-node and localhost traffic respectively. On a single-node cluster, only 6817 (slurmctld) needs to be reachable, and even that only if remote `sbatch` is planned.
- **Fix**: For now open only `6817/tcp`. When a second node is added, restrict 6818/6819 to the cluster subnet, not the world.

### LOW / QUALITY

**14. `gres.conf` shard lines have no `Cores=`** (`gres.conf:56-59`)
- They inherit from the GPU's `Cores=` in current SLURM, but explicit is safer across versions and aids diagnosis.
- **Fix**: Add `Cores=0-15` to shard lines for `/dev/nvidia0,1` and `Cores=16-31` for `/dev/nvidia2,3`.

**15. No CPU-only partition** (`slurm.conf:206`)
- Pure CPU jobs go through `gpu`. They don't consume GPUs, but the partition name is misleading and `--gres` defaults aren't set for CPU work.
- **Fix**: Add `PartitionName=cpu Nodes=enif Default=NO MaxTime=7-00:00:00 DefaultTime=01:00:00 DefMemPerCPU=4096 OverSubscribe=NO State=UP`. Keep `gpu` as default.

**16. `AccountingStorageTRES` doesn't track GPU memory** (`slurm.conf:134`)
- `gres/gpumem` enables per-job VRAM accounting via `sacct`. Useful when a thesis chapter asks "what's our average VRAM utilization?".
- **Fix**: Append `,gres/gpumem` to the TRES list.

**17. No `MailProg` set** (`slurm.conf` — omission)
- Users passing `--mail-user=…` get nothing; SLURM silently ignores the flag.
- **Fix**: Either `MailProg=/usr/bin/mail` (after installing `mailutils`) or document in README that mail isn't wired.

**18. `AutoDetect=nvml` would replace four manual `Name=gpu …` lines** (`gres.conf:35-38`)
- SLURM probes topology from NVML, keeping `Cores=` and `Links=` correct if BIOS NUMA layout or GPU slots ever change. Manual entries freeze today's `nvidia-smi topo -m`.
- **Fix**: At top of `gres.conf`: `AutoDetect=nvml`, then drop the four `Name=gpu` lines (keep shard lines as-is — shards aren't NVML-discoverable).

**19. `HealthCheckProgram` not configured** (`slurm.conf` — omission)
- A wedged GPU driver, ECC error, or stuck process can keep the node `idle` to SLURM while jobs land on it and immediately fail. Health-check would DRAIN proactively.
- **Fix**: Optional. If you install [LBNL NHC](https://github.com/mej/nhc): `HealthCheckProgram=/usr/sbin/nhc` + `HealthCheckInterval=300` + `HealthCheckNodeState=ANY,CYCLE`.

**20. README's quick guide doesn't mention the password env var** (`README.md:39-44`)
- The README still shows `IDENTIFIED BY 'CHANGE_ME_STRONG_PASSWORD'` while `setup.sh` now uses `$SLURM_DB_PASS`. The two have diverged.
- **Fix**: Update the README's step 6 SQL block to reference `$SLURM_DB_PASS` and add a one-liner explaining that `setup.sh` reads it from the environment or prompts.

## Verification

After applying changes:

1. **Static syntax check** — works without root, no daemons started:
   ```bash
   sudo slurmctld -Dcv 2>&1 | head -50    # exits non-zero on syntax error
   sudo slurmdbd   -Dvv 2>&1 | head -30
   ```
2. **Idempotency** — run `bash setup.sh` twice on a clean VM. Second run must exit 0 with no destroyed state (munge key unchanged, DB user unchanged, cluster still registered).
3. **Hardware match** — `sudo slurmd -C` output's `RealMemory`, `CPUs`, `Sockets`, `CoresPerSocket`, `ThreadsPerCore` all match `slurm.conf`. Node should come up `idle`, not `down`.
4. **Epilog** — submit a trivial job (`srun true`); `journalctl -u slurmd | grep -i epilog` shows no errors.
5. **NUMA locality regression test for #4** — submit `srun --gres=gpu:h100:1 -c 16 --cpu-bind=verbose true` repeatedly until it lands on GPU2 or GPU3 (`scontrol show job <id> | grep -E 'GRES|CPU_IDs'`); allocated CPU IDs must all be on socket 1 (cores 16–31). Before the fix, this test occasionally allocates cross-socket; after, it should be socket-local every time.
6. **Shard request with type** — `srun --gres=shard:h100:3 env | grep CUDA_VISIBLE_DEVICES` succeeds (regression check on round-1 fix #5).
7. **MPI** — if keeping pmix: `srun --mpi=list | grep pmix` shows the plugin. If dropping: `MpiDefault=none` and skip.
8. **Interactive timeout** — `srun --pty bash`, idle for 6 min; with `InactiveLimit=1800` the shell is still alive.
9. **Swap absorption** — synthetic `stress-ng --vm 1 --vm-bytes 95% --timeout 30s` inside a job with `--mem=…` close to peak; with `AllowedSwapSpace=10` the job survives transient overshoot, with `0` it OOMs.
10. **Archive dir** — after bumping the system clock past `PurgeJobAfter` (or running `sacctmgr archive dump`), `/var/spool/slurm/archive/` contains a flat file and `slurmdbd.log` shows no archive error.
