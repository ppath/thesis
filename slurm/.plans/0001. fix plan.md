# SLURM Configuration Analysis — `enif` (plast-hpc)

## Context

Single-node SLURM cluster with dual-socket Xeon (2×16c/2t = 64 logical CPUs), ~512 GB RAM, 4 × H100 GPUs. The goal is to find bugs, omissions, and improvements across `slurm.conf`, `gres.conf`, `cgroup.conf`, `slurmdbd.conf`, and `setup.sh`.

---

## Findings

### CRITICAL

**1. Secrets use an ambiguous placeholder with no automated substitution** (`slurmdbd.conf:34`, `setup.sh:34`)
- `StoragePass=CHANGE_ME_STRONG_PASSWORD` is copied verbatim into `/etc/slurm/slurmdbd.conf`. The same literal string is echoed into the MariaDB `CREATE USER` statement in `setup.sh`. There is no mechanism to inject a real password — the operator must manually edit two places, which is error-prone.
- **Fix**: Replace the placeholder in `slurmdbd.conf` with a distinct token (`__SLURM_DB_PASS__`) and update `setup.sh` to:
  1. Read the password interactively or from `$SLURM_DB_PASS` env var.
  2. Copy `slurmdbd.conf` and then `sed -i "s/__SLURM_DB_PASS__/$pass/g" /etc/slurm/slurmdbd.conf`.
  3. Use the same `$pass` variable in the `CREATE USER` SQL statement.
  This way the conf file is safe to commit (a clearly non-functional token), and there is exactly one place to set the real password.

---

### HIGH

**2. `InactiveLimit=0` — idle interactive jobs never killed** (`slurm.conf:135`)
- An `srun --pty bash` session that a user walks away from will hold GPUs/cores forever.
- **Fix**: Set `InactiveLimit=300` (5 minutes) or similar. Users can pass `--time` if they need longer.

**3. No GPU epilog for state cleanup between jobs** (`slurm.conf` — missing)
- A crashed or misbehaving CUDA job can leave GPU memory allocated or NCCL communicators open. The next job on that GPU may fail or see corrupted state.
- **Fix**: Add `Epilog=/etc/slurm/epilog.sh` to `slurm.conf` and create a script that runs `nvidia-smi --gpu-reset -i $SLURM_JOB_GPUS` (or at minimum logs `nvidia-smi` output for the allocated GPUs). Also add `EpilogMsgTime=2000` to give it headroom.

---

### MEDIUM

**4. `PriorityType` not set → defaults to `priority/basic`** (`slurm.conf` — missing)
- `slurmdbd` + associations are fully wired up, which generates fairshare data — but `priority/basic` ignores it. All that accounting data is collected and never used for scheduling.
- **Fix**: Add to `slurm.conf`:
  ```
  PriorityType=priority/multifactor
  PriorityWeightFairshare=100000
  PriorityWeightAge=1000
  PriorityDecayHalfLife=7-0
  ```

**5. `shard` entries in `gres.conf` missing `Type=h100`** (`gres.conf:56-59`)
- The `gpu` lines have `Type=h100`, so users can request `--gres=gpu:h100:1`. But the `shard` lines have no `Type=`, so `--gres=shard:h100:3` (as shown in README examples) will **fail** — only `--gres=shard:3` works.
- The README at line 107 shows `--gres=shard:h100:3` which is broken with the current config.
- **Fix**: Change shard lines to:
  ```
  Name=shard Type=h100 Count=10 File=/dev/nvidia0
  Name=shard Type=h100 Count=10 File=/dev/nvidia1
  Name=shard Type=h100 Count=10 File=/dev/nvidia2
  Name=shard Type=h100 Count=10 File=/dev/nvidia3
  ```

**6. `AccountingStorageTRES` missing `node` TRES** (`slurm.conf:122`)
- Current: `gres/gpu,gres/shard,cpu,mem`. Adding `node` enables per-node usage tracking (important when a second node is added later). Omitting it means node-level reports are incomplete.
- **Fix**: `AccountingStorageTRES=node,cpu,mem,gres/gpu,gres/shard`

---

### LOW / QUALITY

**7. `Waittime=0` is an obsolete parameter** (`slurm.conf:138`)
- This parameter was removed in SLURM 21.08. It causes a warning in newer versions and should be deleted.

**8. Missing `SlurmctldParameters=enable_configless`** (`slurm.conf` — omission)
- The README already documents a second-node expansion path. With `enable_configless`, new compute nodes can pull `slurm.conf` from the controller instead of requiring a manual file copy, which reduces drift.
- **Fix**: Add `SlurmctldParameters=enable_configless`

**9. `setup.sh` uses `$USER` in a sudo context** (`setup.sh:55`)
- `sudo sacctmgr ... add user "$USER"` — if this script is run with `sudo bash setup.sh`, `$USER` is `root`, not the actual operator.
- **Fix**: Use `SUDO_USER` with a fallback: `${SUDO_USER:-$USER}`.

**10. No archival configured in `slurmdbd.conf`** (`slurmdbd.conf` — omission)
- `PurgeJobAfter=12months` hard-deletes records. For a research cluster, old job data may be needed for grant reporting.
- **Fix**: Add archive settings so records are dumped to flat files before deletion:
  ```
  ArchiveEvents=yes
  ArchiveJobs=yes
  ArchiveSteps=yes
  ArchiveDir=/var/spool/slurm/archive
  ```

**11. `MaxRAMPercent=100` may be ignored on cgroup v2** (`cgroup.conf:23`)
- `MaxRAMPercent` is a cgroup v1 knob. On Ubuntu 22.04+ (pure cgroup v2), SLURM's cgroup v2 plugin uses `memory.max` directly from `ConstrainRAMSpace=yes` and `AllowedRamSpace=100`. The parameter is harmless but misleading.
- **Fix**: Remove `MaxRAMPercent=100` (it's redundant with `AllowedRamSpace=100`).

**12. `RealMemory=515000` seems slightly over actual RAM** (`slurm.conf:172`)
- The machine is described as "~512 GB RAM" = 524288 MB physical. With `MemSpecLimit=12288`, schedulable memory would be 515000 − 12288 = 502712 MB ≈ 491 GB — but if `slurmd -C` reports less than 515000 MB available, SLURM will mark the node DOWN due to a `RealMemory` mismatch.
- The setup instructions already say to run `slurmd -C` to verify. Just flagging that `515000` looks a bit high given the stated physical RAM.
- **Fix**: Run `sudo slurmd -C` and adjust `RealMemory` to match (minus a safe OS headroom buffer of ~2 GB).
